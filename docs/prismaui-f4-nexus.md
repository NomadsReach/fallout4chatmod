[size=6][b][color=#FFFFFF]Introduction[/color][/b][/size]
[line]
[/center]

[quote]
[size=4][color=#E1E1E1][i]A modern framework that allows you to build Fallout 4 interfaces using the same technologies that power the web.[/i][/color]

[color=#E1E1E1]Forget the limitations of Scaleform, Adobe Flash, and ActionScript. With PrismaUI F4, you render interfaces directly into the game pipeline using [b][color=#BD3DF8]HTML5[/color][/b], [b][color=#BD3DF8]CSS3[/color][/b], and [b][color=#BD3DF8]JavaScript (ES2022)[/color][/b].[/color]

[color=#E1E1E1]Whether you want to build custom HUDs, complex menus, interactive terminals, or full inventory systems using modern web frameworks like React, Vue, or Svelte, PrismaUI F4 is the solution to create any interface for Fallout 4.[/color][/size]
[/quote]

[size=6][b][color=#FFFFFF]The Future of Fallout 4 UI Modding[/color][/b][/size]
[line]
[/center]

[quote]
[size=4][color=#E1E1E1]Fallout 4's modding scene has long relied on incredible tools like [b]Mod Configuration Menu (MCM)[/b], [b]FallUI[/b], and other mods built on Scaleform's SWF format. These projects pushed the limits of what was possible and the community owes them a great deal.[/color]

[color=#E1E1E1]But Scaleform is a [b]discontinued Flash-based technology from 2011[/b]. SWF files require specialized legacy tools to create, cannot be hot-reloaded, offer no debugging, and are increasingly difficult to maintain as the technology falls further behind. Talented mod authors are spending more time fighting the toolchain than building the experiences they want to create.[/color]

[color=#00ff88][b]PrismaUI F4 changes that.[/b][/color]

[color=#aaffcc]Instead of Flash, you write HTML, CSS, and JavaScript -- the same languages powering every website on the internet. Instead of Adobe Animate, you use VS Code. Instead of a SWF binary, your UI is a plain .html file you can edit and reload in seconds. You get the full modern web toolchain: React, Vue, Svelte, Tailwind, npm packages, and a real WebKit DevTools inspector.[/color]

[color=#aaffcc]This is not a wrapper around Scaleform. PrismaUI F4 bypasses it entirely, embedding a real HTML renderer directly into the game process. Any plugin built on PrismaUI F4 carries zero dependency on MCM, FallUI, or any SWF infrastructure.[/color]

[color=#ffff88]This is the direction Fallout 4 UI modding is heading. We are building it here, in the open, for the community.[/color]
[/quote]

[size=6][b][color=#FFFFFF]Simple and Powerful API[/color][/b][/size]
[line]
[/center]

[quote]
[size=4][color=#E1E1E1]The entire API surface is a single header file: [b][color=#BD3DF8]PrismaUI_F4_API.h[/color][/b][/color]
[color=#E1E1E1]No import libraries. No extra dependencies. Drop it in your source folder and you are ready.[/color][/size]

[spoiler=Quick Start Code Example]
[code]
#include "PrismaUI_F4_API.h"

static PRISMA_UI_API::IVPrismaUI2* g_api  = nullptr;
static PrismaView                   g_view = 0;

static void OnDomReady(PrismaView view)
{
    g_api->RegisterJSListener(view, "onClose", [](PrismaView, const char*) {
        g_api->Unfocus(g_view);
        g_api->Hide(g_view);
    });
}

static void F4SEMessageHandler(F4SE::MessagingInterface::Message* msg)
{
    if (msg->type == F4SE::MessagingInterface::kGameDataReady) {
        g_api = PRISMA_UI_API::RequestPluginAPI<PRISMA_UI_API::IVPrismaUI2>();
        if (!g_api) logger::error("PrismaUI_F4 not found!");
    }
    if (msg->type == F4SE::MessagingInterface::kPostLoadGame) {
        if (g_api && g_view == 0) {
            g_view = g_api->CreateView("mymenu.html", OnDomReady);
            g_api->Hide(g_view);
        }
    }
}

// Push data from C++ into your page at any time
g_api->Invoke(g_view, "updateAmmo(24, 120)");
[/code]
[/spoiler]

[size=6][b][color=#FFFFFF]Performance Notes[/color][/b][/size]
[line]
[/center]

[quote]
[size=4][color=#E1E1E1]PrismaUI F4 uses CPU rendering. Each view is rendered off-screen and composited as a D3D11 texture above the game world.[/color]

[list]
[*] UI refresh rate is capped at 60 FPS
[*] Keep views hidden when not in use -- hidden views are not rendered
[*] Prefer flat colours and simple transitions over heavy gradients, blur filters, or large box-shadows
[*] Use a single view with internal JS routing rather than many separate views where possible
[/list][/size]
[/quote]

[size=6][b][color=#FFFFFF]Security & Privacy[/color][/b][/size]
[line]
[/center]

[quote]
[size=4][color=#E1E1E1]We know this question comes up for any mod that ships a DLL. Here is exactly what PrismaUI F4 does and does not do.[/color]

[b][color=#BD3DF8]What it is:[/color][/b]
[list]
[*] [b]A standard F4SE plugin[/b] -- It loads the same way every other F4SE plugin does -- through F4SE's plugin interface, with no elevated privileges.
[*] [b]Open Source[/b] -- Under the MIT License. Every line of code is auditable. The source is public. There are no hidden components.
[*] [b]An HTML renderer, not a browser[/b] -- Ultralight is an embedded rendering engine -- it has no network stack and makes no outbound connections. Your HTML pages load from local files on your hard drive only.
[*] [b]Isolated to the game process[/b] -- The Ultralight renderer runs inside Fallout4.exe. It does not spawn background processes, install services, or write outside the game's data folders.
[/list]

[b][color=#ff6060]What it does not do:[/color][/b]
[list]
[*] No internet connections -- ever. HTML views load from local files only.
[*] No telemetry, analytics, or data collection of any kind.
[*] No background services, scheduled tasks, or anything that runs outside the game.
[*] No writes to your system outside of standard F4SE log files.
[*] No dependency on external servers or third-party services.
[/list]

[color=#ffff88]If you are still uncertain, the source code is there to read. We welcome scrutiny.[/color][/size]
[/quote]
